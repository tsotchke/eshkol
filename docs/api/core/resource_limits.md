### `eshkol_track_allocation`

*Function* ‚Äî line 179

```c
bool eshkol_track_allocation(size_t bytes);
```

Register a heap allocation of `bytes` against the tracked usage total (call from the arena allocator before/as it allocates). Thread-safe (lock-free CAS loop). If the new total would exceed `max_heap_bytes` (when non-zero) or overflow accounting, records ESHKOL_LIMIT_HEAP_HARD as the last limit error; if `enforce_hard_limits` is set this additionally logs an error and requests a runtime interrupt with ESHKOL_SHUTDOWN_MEMORY, and the call returns false. Also updates peak usage and, once usage crosses the soft-limit threshold, logs a one-time warning (if `enable_warnings` is set) and records ESHKOL_LIMIT_HEAP_SOFT.

**Parameters**

- `bytes` ‚Äî Number of bytes being allocated (0 is always allowed).

**Returns**

true if the allocation is allowed to proceed, false if it would exceed the hard limit.

### `eshkol_stack_push`

Enter a new tracked stack frame (call at function entry). Increments a thread-local depth counter. If the new depth exceeds `max_stack_depth`, records ESHKOL_LIMIT_STACK_OVERFLOW as the last limit error, decrements the counter back down (the disallowed frame is not counted), and ‚Äî if `enforce_hard_limits` is set ‚Äî logs an error. ### `eshkol_start_timer`

Start (or restart) the execution timer. Records the current time as the timer's start point and, if `enforce_hard_limits` is set and the effective timeout is non-zero, detaches a background thread that sleeps for the timeout duration and then ‚Äî unless the timer was stopped or restarted in the meantime (tracked via a generation counter) ‚Äî records ESHKOL_LIMIT_TIMEOUT and requests a runtime interrupt with ESHKOL_SHUTDOWN_TIMEOUT. eshkol_is_timed_out() additionally performs its own polling-based check, so timeout enforcement does not rely on the watchdog thread alone.

**Parameters**

- `timeout_ms` ‚Äî Timeout override in milliseconds; 0 uses the currently configured `max_execution_time_ms` limit instead.
