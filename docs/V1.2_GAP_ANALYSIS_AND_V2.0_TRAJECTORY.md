# Eshkol — v1.2-scale gap analysis and v2.0-quantum trajectory

Generated 2026-05-08 from a full codebase ICC sweep + complete docs read.
Purpose: distinguish what v1.2 actually delivers from what its docs claim;
lay out a concrete sequence to v2.0 with each gap pinned to file:line.

---

## TL;DR

v1.2-scale is **substantially shipped** (438 tests, 100% pass; ASan/UBSan/TSan
clean; 16-lane CI matrix) but four classes of gap live in the tree right now:

1. **Silent-correctness bug**: conv2d / batchnorm / layernorm / attention
   *backward* passes are missing the second-operand gradient wiring
   (`node->input2`). Training relies on these. Worst gap on the list.
2. **Three synthetic stubs** that the oracle infrastructure flags but the
   release notes don't — `vm_macro_match`, `codegenTailCallFromContext`,
   `parseAllAstsFromString`. Each is small and well-localized.
3. **v1.3 features that cmake/parser haven't started**: `define-library`,
   `~{expr}` string interpolation, `#:keyword` args, `let-match`. ROADMAP
   commits to all four for June 2026.
4. **Trace-evidence vacuum**: every ICC oracle is currently BLOCKED on
   missing `scripts/icc_traces/*.jsonl` records. The fix is to run
   `scripts/run_icc_smoke.sh` and `scripts/run_sdnc_oracle.sh` — no code
   change. Five oracles flip green from one harness invocation.

Plus a cleanup tail: **570 dead-code symbols** (mostly llvm_codegen.cpp
migration leftovers) and **17 shell-hardening findings** in build scripts.

The path to v2.0-quantum is well-signposted in `ROADMAP.md` and
`COMPILER_ROADMAP.md`. Nothing in v1.3-v1.7 needs new design; v1.8 platform
program is in `docs/platform/` (governance-ready, code-not-merged); v1.9
dependent-types extends the existing HoTT type infra; v2.0 quantum +
Lean integration is the only stretch that doesn't yet have a written design.

## 1. Material gaps in v1.2-scale (must fix or document)

### 1.1 AD input2 backward plumbing — silent wrong gradients

**Where:** `lib/bridge/tensor_backward.cpp:9-10` (TODO comment), gates at
backward kernels that check `if (node->input2) ...`. Same shape on the
GPU side (`lib/backend/gpu/gpu_memory_cuda.cpp:762-774` declares
`cuda_conv2d_backward_input_f64`, `cuda_conv2d_backward_kernel_f64`,
`cuda_batchnorm_backward_f64`, `cuda_layernorm_backward_f64` and gates
their use on `input2` being populated).

**Why it's the worst gap:** users call `(gradient model loss)` and the
compiler emits valid IR, the run completes, and the gradient w.r.t. the
SECOND operand of conv/batchnorm/layernorm is silently zero (or stale).
Single-operand ops (matmul-by-scalar, elementwise) work; ops where the
gradient must split across two inputs do not. That includes every modern
neural-network primitive.

**Listed in `RELEASE_NOTES.md` carry-forward** as a v1.3 first-week item
("AD `input2` plumbing for non-matmul tensor ops"). It's labelled
correctly — it is just shipped quietly enough that downstream consumers
(Noesis, Selene) might not realize their training updates are broken.

**Action**: a release-blocker for v1.3-evolve. Fix and add a regression:
- forward node populates `input2` for every two-operand op
- backward kernel reads `input2`, accumulates the second gradient
- test: numerical-gradient check against finite differences for conv2d,
  batchnorm, layernorm, attention — should agree to 1e-6

**Estimated effort**: 3-5 days. Mechanical work; the kernels exist, just
not wired.

### 1.4 XLA JIT cache missing

`lib/backend/xla/xla_compiler.cpp:115-117`:
```cpp
// TODO: XLA JIT cache — wire into compilation pipeline when StableHLO is active.
// TODO: XLA JIT cache — implement with llvm::parseIR() for StableHLO mode.
```

XLA mode currently recompiles each module every invocation. For an XLA
benchmark that compiles once and runs millions of times this is fine —
but for production training loops where shapes change and recompilation
fires regularly, this is a real perf cliff. Listed in v1.2 release notes
as XLA fusion activation; the cache itself is unfinished.

**Action**: v1.3 follow-up. Not on the critical path because XLA is
opt-in via `-DESHKOL_XLA_ENABLED=ON`.

### 1.7 Build-script shell hardening

ICC `production-audit` flags 17 findings in `scripts/build-*.sh`:
- 2 HIGH: `direct_variable_heredoc` (heredocs that interpolate
  unquoted vars; injection if env is hostile)
- 9 MEDIUM: `variable_output_redirect` (output redirection through
  unquoted vars)
- 6 MEDIUM: `variable_rm` (rm targets through unquoted vars)

Not security-critical for trusted developer machines, but worth fixing
for CI hygiene and any embedded-target build flow that runs as root.
A 2-hour pass with `shellcheck` would close most of them.

## 4. Honest deferments (recommended)

Some items in `ROADMAP.md` should drop to "best-effort" or move
explicitly to v2.x to keep monthly releases shippable:

- **Multi-GPU dispatch** (currently scheduled v1.4): the cost-model
  rewrite for device selection is a 2-week project unto itself.
  Recommend v1.4.x or v1.5.
- **Vulkan compute** (currently v1.8): cross-platform GPU beyond
  Metal/CUDA is ~3-4 weeks. Could split into Vulkan-on-Linux first
  (v1.8) and Vulkan-everywhere (v1.8.x).
- **Neural theorem provers** (v1.7): combines v1.5 embeddings + v1.6
  backward chaining + new neural heuristic — bump to v1.7.x.
- **Lean integration** (v2.0): linking against `libleanc.a` /
  `libleanshared.dylib` is feasible (toolchain installed) but the
  elaborator API is heavy. Could ship a subprocess-based v0.1
  in v1.5 (Noesis-friendly) and the embedded version in v2.0.
- **Distributed computing** (v1.8): multi-node AllReduce / gRPC /
  MessagePack — the largest remaining roadmap item not yet costed.
  Recommend a separate v1.8.5 / v1.9.5 milestone.

These aren't commitments to slip — they're invitations to be explicit
about which items drive the headline release vs. which trail behind.
