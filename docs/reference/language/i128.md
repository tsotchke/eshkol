# Native 128-bit Integers (`i128`)

`i128` is a **distinct, fixed-width, two's-complement signed integer type** —
128 bits wide, value range **−2¹²⁷ … 2¹²⁷−1**. It is deliberately **off the
numeric tower**: it does not participate in fixnum→bignum→rational→real→complex
promotion, and it is **not** produced or consumed by the ordinary numeric
operators (`+`, `*`, `<`, …). Every i128 operation has its own `i128-` builtin,
and every crossing to or from the tower is an **explicit conversion**.

The defining difference from a bignum: **bignums never overflow** (they grow),
whereas **i128 arithmetic wraps** modulo 2¹²⁸ (two's complement). This makes
i128 the right tool when you want deterministic machine-word semantics — hashing,
checksums, fixed-width bit manipulation, or an exact 128-bit accumulator — rather
than unbounded exact integers.

> Behaviour below is the **native code path** (the LLVM JIT used by `-r`, and
> AOT). The bytecode VM implements the identical value semantics; the one
> documented difference is error handling (see [Errors](#errors)).

## Representation

An i128 is **heap-boxed**: the tagged-value data union is 64 bits, so a 128-bit
integer cannot be an immediate. It is stored as an 8-byte object header followed
by a 16-byte little-endian `{lo, hi}` payload, under heap subtype
`HEAP_SUBTYPE_I128` (25). The payload layout matches the two-u64 FFI marshalling
contract used elsewhere in the runtime, so a boxed i128 is directly interoperable
with foreign 128-bit values. It is a flat POD leaf with no interior pointers, so
it is copied correctly by the region evacuator without a deep walk.

## Constructors and predicate

| Form | Meaning |
|------|---------|
| `(i128 x)` | Box an exact fixnum `x` as an i128. |
| `(int->i128 x)` | Alias of `(i128 x)`. |
| `(string->i128 s)` | Parse a decimal string over the **full** signed range, including `-2^127`. |
| `(i128? x)` | `#t` iff `x` is an i128. |

```scheme
(display (i128? (i128 5)))   (newline)   ; => #t
(display (i128? 5))          (newline)   ; => #f  (a fixnum is not an i128)
(display (string->i128 "170141183460469231731687303715884105727")) (newline)
```
```
#t
#f
170141183460469231731687303715884105727
```

## Wrapping arithmetic

`i128-add`, `i128-sub`, `i128-mul`, and `i128-neg` all wrap two's-complement at
the ±2¹²⁷ boundary — they never promote to bignum.

```scheme
(define MAX (string->i128 "170141183460469231731687303715884105727"))   ; 2^127 - 1
(display (i128->string (i128-add MAX (i128 1)))) (newline)  ; wraps to -2^127
(display (i128->string (i128-mul MAX (i128 2)))) (newline)  ; wraps to -2
```
```
-170141183460469231731687303715884105728
-2
```

`(i128-neg n)` negates with wrap: `(i128-neg I128_MIN)` is `I128_MIN` again,
because `+2^127` is not representable.

## Shifts

`i128-shl` (left), `i128-ashr` (arithmetic / sign-preserving right), and
`i128-lshr` (logical / zero-fill right) take a **shift count in `0…127`**. A count
outside that range raises (native) / is fatal (VM).

```scheme
(define MIN (string->i128 "-170141183460469231731687303715884105728"))
(display (i128->string (i128-shl (i128 1) 127))) (newline)   ; == I128_MIN
(display (i128->string (i128-lshr MIN 127)))      (newline)  ; zero-fill  => 1
(display (i128->string (i128-ashr MIN 127)))      (newline)  ; sign-fill  => -1
```
```
-170141183460469231731687303715884105728
1
-1
```

## Comparisons

`i128=?`, `i128<?`, `i128>?`, `i128<=?`, `i128>=?` compare two i128 values and
return a boolean.

```scheme
(display (i128<? (i128 -1) (i128 1))) (newline)   ; => #t
(display (i128=? (i128 5) (i128 5)))  (newline)   ; => #t
```

## Division

`i128-quotient` and `i128-remainder` are **truncated** (C semantics): the
quotient truncates toward zero and the remainder carries the sign of the
dividend. Division by zero raises (native) / is fatal (VM).

```scheme
(display (i128->string (i128-quotient (i128 -7) (i128 2)))) (newline)  ; => -3
(display (i128->string (i128-remainder (i128 -7) (i128 2)))) (newline) ; => -1
```

## Conversions and printing

| Form | Meaning |
|------|---------|
| `(i128->string n)` | Decimal string (the exact inverse of `string->i128`). |
| `(i128->int n)` | Narrow to a fixnum; **raises if `n` is outside the fixnum range**. |

`display` and `write` both render an i128 as a plain decimal number.

```scheme
(display (string->i128 "-170141183460469231731687303715884105728")) (newline)
```
```
-170141183460469231731687303715884105728
```

## Errors

The following conditions are errors:

- `i128-quotient` / `i128-remainder` with a zero divisor,
- a shift count outside `0…127`,
- `i128->int` on a value that does not fit in a fixnum,
- `string->i128` on a non-numeric string or a value outside `[-2^127, 2^127-1]`,
- passing a non-i128 (and non-fixnum) to an `i128-` operation.

On the **native path** these raise catchable R7RS conditions, so `guard` /
`with-exception-handler` observe them. On the **bytecode VM** they are fatal
and halt with a diagnostic — matching the VM's existing behavior for integer
division-by-zero and `(error …)`.

```scheme
(display (guard (e (#t 'caught))
           (i128-quotient (i128 1) (i128 0)))) (newline)   ; native => caught
```

## Notes and future work

- i128 is intentionally kept off the auto-promotion path so ordinary integer
  code is unaffected; mixing an i128 with a tower number is never implicit.
- The arithmetic `i128-` builtins also accept a plain fixnum operand (its exact
  value, **not** a tower promotion), so idioms like `(i128-add x 1)` are ergonomic.
- This release boxes i128 on the arena for correctness. An unboxed LLVM-`i128`
  hot path (keeping the value in a register across an arithmetic chain, boxing
  only at type boundaries) is a planned optimization and does not change any of
  the semantics above.

See also: [numeric-tower.md](numeric-tower.md) — i128 is a *distinct fixed-width
type off the tower*, contrasted there with the auto-promoting bignum rung.
