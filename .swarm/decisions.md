# Eshkol Swarm Decisions — v1.3-evolve

| Date | Decision | Reason |
|---|---|---|
| 2026-05-27 | Use a `.swarm/` ledger mirroring QGTL's structure (`tasks/`, `claims/`, `events.jsonl`, `decisions.md`, `risks.md`, `status.md`). | Cross-project consistency lets the same codex-main supervisor pattern run on Eshkol; the QGTL pattern is proven (47 done tasks, readiness 100/0 risks). |
| 2026-05-27 | The ICC `v1.3-evolve` completion oracle is the success gate. Every task's `verification` block invokes `scripts/run_icc_smoke.sh` and asserts the named probe flips to PASS. | Mechanically checkable acceptance prevents "looks done in the diff but the oracle still WARNs" drift. |
| 2026-05-27 | Multi-shot `call/cc` will be implemented via **stack copying** (Chez/Gambit-style), not via heap-allocated activation frames. | Compatible with Eshkol's existing native function ABI via LLVM; no full codegen rewrite. ~3-4w of careful work. |
| 2026-05-27 | Multi-shot `call/cc` **forbids capture inside `gradient` / `derivative`** for v1.3. Runtime trap with a clear error; revisit in v1.5 where AD is being restructured for soft unification anyway. | Cleanest semantics; preserves capture cost. AD-tape interaction is the genuinely hard sub-problem and v1.3 should not block on it. |
| 2026-05-27 | Captured continuations **pin all OALR regions live in the captured stack** via a region-refcount bump on capture, decrement on continuation reclamation. | Prevents use-after-pop when a captured continuation outlives its enclosing region. |
| 2026-05-27 | `(the-environment)` is a **compile-time special form** that emits `eshkol_make_local_env(name1, val1, name2, val2, ..., parent_env)` enumerating every lexically-visible binding at that point + a pointer to `interaction-environment` as the parent. | Eshkol's compiler already tracks lexical scope at every parse point; spilling the local frame to the heap on demand is straightforward. Matches MIT Scheme semantics (locals + globals visible). |
| 2026-05-27 | Source-span stack traces thread `(file, line, col, span-end-line, span-end-col)` through the runtime exception path as a side band on every Eshkol-level call. The format is a per-thread ring buffer of `(span, function-name, locals?)`, capped at 256 frames by default. | Native unwinding via DWARF gives C frames, not Eshkol frames. A side-band stack is the standard approach (Cython, Julia, all dynamic-on-native languages). Capping at 256 prevents pathological growth. |
| 2026-05-27 | Mesh node assignment: platform-native work to its native node. macOS CoreGraphics → atlas/enki. Linux libpng/libjpeg/libwebp → cosbox or old-donkey. Windows GDI+ → jack-blupc. CUDA-side verification of AD `input2` → xavier. | The QGTL pattern showed that platform-native CI on the actual platform reveals bugs Docker-on-Mac misses. |
| 2026-05-27 | Tsotchke-chan submissions use **distinct backends per task** based on task character: Claude for architectural / context-heavy tasks, Codex for tight CRUD work, Qwen/Kimi for high-volume mechanical tasks (e.g. each platform of stb-replacement). | Each backend has different strengths; matching task character lets the swarm move faster than monolithic-backend dispatch. |
| 2026-05-27 | **Default implementation backend is `codex`**, not `claude`. The current `--backend claude` route is a sandboxed advisory workspace with no filesystem access to the Eshkol tree — every claude-dispatched ESH task failed with "this sandboxed advisory workspace cannot read /Users/tyr/Desktop/eshkol/...". `codex` has filesystem access and works correctly. Reserve `claude` for advisory-only review tasks until a filesystem-enabled Claude Code variant is wired. See `.swarm/runtime/dispatch_log.md` for evidence. | Reported to user as an infrastructure issue to fix. |
| 2026-05-27 | `--backend qwen` is **not** a registered backend name; tsotchke-chan maps it silently to `local-agent`. Supported names: `codex, claude, eshkol-agent, local-agent, kimi-agent, openrouter`. | Spec `preferred_backend: qwen` entries should be updated to `local-agent` or `kimi-agent`. |
| 2026-05-27 | `ESHKOL_VALUE_HANDLE` lifecycle states `{open, borrowed, closed}` are runtime-tracked in v1.3 (warning-mode linear check). Compile-time enforcement waits for v1.9 dependent linear types. | Prebuilds v1.4 socket safety while not blocking on the v1.9 type-system work. |
| 2026-06-30 | The v1.3 SICP claim means 100% of the entire book, not a representative subset. ESH-0005 is a release blocker until all manifest-required systems pass under both JIT and AOT: SICP 3.3.4 digital circuits, 3.3.5 constraints, 3.4 concurrency/serializers, 4.2 lazy evaluation, 4.4 query evaluation, 5.3 storage/GC, 5.4 explicit-control evaluation, and 5.5 compilation. ESH-0034 provides the section/example coverage manifest proving no book content is silently omitted; beyond-book stress remains part of the gate. | Prevents a green sprint while any book system or manifest coverage is still absent, and gives parallel agents explicit task boundaries for the remaining work. |
